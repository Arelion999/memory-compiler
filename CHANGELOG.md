# Changelog

Semantic versioning: major.minor.patch. Versions below 1.0 were development milestones (v8-v12 pre-release).

## v1.37.0 — 2026-07-19

Многометочность для взаимозаменяемых статей проверена и **отвергнута**. Кода не касается — фиксируется отрицательный результат и, что важнее, причина, по которой правило нельзя вывести из этих данных.

### Changed

- **Идея.** 32% промахов (23 из 72) имеют на первом месте статью, которая сама записана «правильным ответом» для ДРУГОГО запроса. Выглядит как несправедливость метрики: возвращается равноценная статья, засчитывается ошибка.
- **Требование к правилу.** Расширение множества правильных ответов надувает метрику почти по построению, поэтому правило обязано быть **чисто поведенческим**. Обращаться к модели похожести нельзя: тогда поиск оценивался бы его же представлением о близости.
- **Правило «один и тот же запрос в разное время»** (стандартная агрегация кликов в IR) — **отвергнуто за отсутствием материала**: повторяющихся пар «запрос-проект» в наборе **ноль**, 129 уникальных из 129 записей, чинится 0 промахов из 72. Дубликаты, виднелись раньше (запрос `secret` дважды), были ТЕГОВЫМИ и ушли вместе с `search_by_tag` в v1.36.0. Настоящие поисковые фразы дословно не повторяются.
- **Правило «классы эквивалентности по со-открытию»** — **отвергнуто по цене и по существу.** Чинит 6 промахов из 72, раздувая множество ответов с 1.40 до 3.14 на запрос (×2.25). MRR 0.5512 → 0.5925 (+0.0414), но вклад шести честных починок ≈ +0.0233 — то есть **44% прироста подарок от расширения, а не измерение.** Структурный изъян важнее цифр: объединение **транзитивно** — один запрос с пятью открытиями склеивает пять статей, классы сцепляются через общие, и получается класс из 11 статей, взаимозаменяемыми не являющихся. Это цепочка совместных просмотров, а не эквивалентность.
- Как ориентир абсурда мерилось и заведомо расширительное правило «засчитывать всё, что было ответом где-либо»: чинит 23, раздувает в **95 раз**.

### Chore

- **Куда смотреть вместо этого.** Если статьи действительно дублируют друг друга (`secret_доступы_zarina-khv_ru` против `secret_полные_доступы_zarina-khv_ru`), лечить надо КОРПУС, а не метрику — для этого есть `consolidate`. Но подбирать кандидатов на слияние по со-открытию нельзя ровно по той причине, по которой отвергнуто второе правило.
- Деплой **N/A**: изменение только в `CHANGELOG.md`, поведение кода не затронуто.

## v1.36.0 — 2026-07-19

Взялся за «окно по времени» — и оказалось, что оно лечило симптом. Настоящая причина была в составе источников набора, а критерий их отбора был метрико-ориентированным, то есть круговым.

### Fixed

- **`search_by_tag` убран из источников golden-набора: он НЕ РАНЖИРУЕТ вообще.** Он перебирает файлы, точно сверяет тег в строке `**Теги:**` и возвращает все совпадения в порядке обхода каталога (`handlers.py`) — «первого места» там не существует, поэтому выбор статьи из такого списка ничего не говорит о качестве ранжирования. Клики оттуда меряли не тот конвейер, который оценивается. Данные подтверждают категориальную разницу (299 пар живого лога):

  | источник | пар | запросов | медиана разрыва | событий между | слов в запросе |
  |---|---|---|---|---|---|
  | `ask` | 64 | 41 | 10 с | 0 | 26 |
  | `search` | 118 | 70 | 109 с | 1 | 6 |
  | `get_context` | 10 | 5 | 145 с | 2 | 13 |
  | **`search_by_tag`** | **107** | **22** | **2208 с** | **43** | **1** |

  Тег даёт 4.9 открытия на запрос против 1.7 у поиска — это просмотр списка, а не поиск одной вещи.
- **Эффект больше, чем кажется по числу запросов: набор ужался со 132 до 129, а MRR вырос 0.5212 → 0.5512, recall@1 0.3939 → 0.4419.** Причина — механизм, о котором предупреждал комментарий v1.29.0: источник не просто добавляет пары, он **ПЕРЕНАЗНАЧАЕТ клики.** Теговые «запросы» закрывали настоящие поисковые и забирали их открытия себе; убрали — открытия вернулись владельцам. Это правка ХАРНЕССА, а не улучшение поиска: код ранжирования не тронут.
- ⚠️ **Исправлен сам критерий отбора источников.** Было: «на замере набор становится лучше» (так в v1.29.0 приняли `search_by_tag`) — это отбор по метрике, то есть ровно то круговое рассуждение, которого метрика должна избегать. Стало: **источник годится, только если он вызывает измеряемый конвейер.** `search`, `ask`, `get_context` идут через `_whoosh_async` → `whoosh_search`; `search_by_tag` не идёт. Критерий проверяется по коду, а не по цифре.

### Added

- **Параметр `max_gap_s` в `build_golden`/`load_golden` — окно по времени, по умолчанию ВЫКЛЮЧЕНО.** Замер после удаления тегов: 105 пар из 192 укладываются в 30 с, но 37 пар превышают полчаса и имеют медиану **13 событий** между запросом и открытием (то есть открытие относится уже к другой работе). Естественного разлома в распределении **нет**, а окно заметно двигает цифру (1800 с → MRR 0.5976, 60 с → 0.5999) — поэтому включать его сейчас значило бы выбрать порог по росту метрики. Параметр оставлен, чтобы будущий замер мог проверить окно, не переписывая сборку; менять дефолт — только по критерию, не связанному с итоговой цифрой.

### Chore

- +6 тестов (537 всего): тег не порождает запрос; тег не перехватывает клик у настоящего поиска; окно отсекает поздние открытия; по умолчанию окно выключено; битая отметка времени не роняет пару; служебное поле окна не утекает наружу. Тест `test_golden_accepts_tag_search`, закреплявший прежнее решение, удалён — он кодировал отменённый критерий.
- Деплой **N/A** по поведению: `retrieval_eval` в рантайме не вызывается, поиск не затронут.

## v1.35.0 — 2026-07-19

Отрицательный результат по последнему неперебранному механизму ранжирования. Кода не касается — только документация, чтобы следующий заход не открывал этот вопрос в третий раз.

### Changed

- **Cross-encoder реранк проверен на БОЛЬШОМ пуле и снова отвергнут — вывод v1.27.0 усилился.** Тогда его выключили с оговоркой: «польза берётся из переранжирования большого пула (50–100), при пуле 10 спасать нечего, а поднять пул на этом CPU невозможно». После v1.31.0 пул стал 100, предпосылка отпала — вопрос переоткрыт честно. Замер на очищенном наборе (n=132, `bge-reranker-v2-m3`, 20-ядерный ПК):

  | Конфигурация | MRR | recall@1 | recall@5 | recall@10 | лучше/хуже | цена |
  |---|---|---|---|---|---|---|
  | без реранка | **0.5151** | **0.3939** | **0.7045** | **0.7500** | — | — |
  | реранк топ-20, превью | 0.4769 | 0.3485 | 0.6288 | 0.7576 | +30 / −33 | 8.0 с/запрос |
  | реранк топ-20, полное тело | 0.5073 | **0.3939** | 0.6894 | 0.7424 | +27 / −28 | 22.2 с/запрос |

  На превью (`title`+`preview`, обрез 400 символов — ровно то, что скармливает `rerank()`) реранкер **активно вредит**: −0.038 MRR, вдвое выше порога различимости. С полным телом вред исчезает (−0.008, в пределах шума; recall@1 совпадает с базой ровно), но **выигрыша нет**: +27/−28 по запросам — идеальный ноль при 22 секундах на запрос. Гипотеза «реранкеру нечего читать» подтвердилась наполовину: обрезанное превью объясняет ВРЕД, но не отсутствие пользы. Глубина 50 не гонялась осознанно — направление уже показало себя, а стоило бы это часа.

- ⚠️ **Остаток качества — на 38% артефакт разметки, а не промахи поиска.** Разбор всех 80 запросов, где цель не первая: служебные файлы на первом месте лишь **2 раза**, статей чужих проектов — **0**, то есть мусора в выдаче нет и первой почти всегда стоит осмысленная статья. При этом **23 из 80 (29%)** — статья, которая сама записана «правильным ответом» для другого запроса (пользователь открывал обе по схожему поводу), а **17 из 80** — запросы из одного-двух слов (`secret`, `deploy`, `доступы`, `ssh`), у которых единственно верного ответа не существует в принципе. Показательная пара: на «SSH доступы zarina-khv.ru» первой идёт `secret_полные_доступы_…`, а ожидается `secret_доступы_…`; на запросе «доступы» те же две статьи меняются местами. Косинус «первая ↔ ожидаемая» при этом почти не отличается от случайной пары внутри проекта (0.922 против 0.905) — near-дублей по содержанию НЕТ, дедупликацией не выиграть.

### Chore

- **Механизмы ранжирования исчерпаны, узкое место — набор.** Перебраны веса каналов RRF, бусты полей, отсечка, порядок слияния, cross-encoder; значимого плюса не дал ни один. Дальше работать следует не над алгоритмом, а над оценкой: окно по времени в `build_golden` (открыто с v1.32.0) и честная многометочность — засчитывать любую из взаимозаменяемых статей. Это уберёт фальшивую часть потолка и поднимет мощность замера с нынешних ~0.024 MRR.
- Деплой **N/A**: изменения только в `CHANGELOG.md` и `README.md`, поведение кода не затронуто.

## v1.34.0 — 2026-07-19

Локальное зеркало `knowledge/` три недели отставало от NAS, и причина оказалась не в настройках синхронизации.

### Added

- **`scripts/mc-drive-nudge.sh` — обход того, что Synology Drive не видит записи контейнера.** Синхронизация папки **двусторонняя и исправная**: замер зондами показал, что файлы ходят за 20–60 с в обе стороны (создание на ПК → NAS за секунду; удаление на ПК → пропало с NAS за 45 с; создание на NAS в корне → на ПК за 20 с, в подпапке → 40 с, через переименование из дотфайла → доехало). Но **ни одна статья, записанная контейнером, вниз не доехала** — он пишет через bind-mount, и механизм уведомлений Drive такие изменения не ловит. Итог: 480 статей локально против 1719 на NAS. Скрипт периодически обновляет отметку времени у файлов, изменившихся с прошлого прогона — этого события Drive достаточно. Ставится через Планировщик заданий DSM от root (у `UserAI` нет sudo, `/etc/crontab` не пишется), рядом с `mc-watcher.sh`.

### Fixed

- **Ловушка, которая едва не стоила 480 статей боевой базы.** Отставшее зеркало выглядит мёртвым остатком, и напрашивается «прибраться» через проводник. Но синхронизация живая: удаление уезжает на NAS. Проверено зондом до, а не после — файл, удалённый локально, исчез с NAS за 45 секунд. Разовое лечение отставания (`find … -exec touch {} +` на NAS) вернуло зеркало к 1719 статьям за 83 секунды. Отдельная тонкость: статьи, переехавшие между проектами, остаются локально по СТАРЫМ путям — после дозаполнения их оказалось 33 «лишних» (1752 против 1719). Удалять их безопасно ровно потому, что на NAS таких путей нет, но это проверено поимённо, а не принято на веру.

- **`.gitattributes` с `eol=lf` — все `*.sh` в репозитории уезжали на NAS с CRLF и не запускались.** Обнаружено при первом же прогоне нового скрипта на NAS: `syntax error: unexpected end of file`. Ловушка тихая и общая: в самом git блоб правильный (LF), ломается только рабочее дерево, потому что глобальный `core.autocrlf=true` подменяет переводы строк при выкладке, а на NAS уезжает именно рабочее дерево через SynologyDrive. Локально это не ловится — bash из Git Bash под Windows CRLF терпит, а bash на NAS нет. На момент находки CRLF были во ВСЕХ четырёх скриптах, включая `release.sh` и `mc-watcher.sh`. Правило добавлено для `*.sh`, `*.py`, `Dockerfile`, `*.yml`; дерево перенормализовано.

### Chore

- Деплой **N/A**: изменение только в `scripts/`, в контейнер этот каталог не монтируется. Поведение сервера не затрагивается.

## v1.33.0 — 2026-07-19

Первая осмысленная проверка весов: до v1.31.0 мерить их было нельзя (канал BM25 пустовал на половине запросов), теперь можно. **Улучшений нет — дефолты сохранены.** Зафиксировано, чтобы следующий заход не начинал с тех же гипотез.

### Added

- **Веса каналов в RRF вынесены в env** (`RRF_WEIGHT_BM25`, `RRF_WEIGHT_SEMANTIC`, оба `1.0`). Значение имеет только отношение весов — общий множитель уходит в масштаб скора. Ручка нужна не ради текущего результата, а чтобы перезамер при большем наборе стоил минуту.

### Changed

- **Гипотеза «бусты полей» проверена наконец по-настоящему — и не подтвердилась.** В v1.31.0 её отвергли по косвенному признаку (живой BM25 лишь у половины запросов); теперь он есть у всех, и остаток целиком лежит там, где веса и работают: 56.8% запросов имеют цель в пуле, но не первой. Сетка query-time бустов (`MultifieldParser(fieldboosts=...)`, пересборка индекса не нужна) на очищенном наборе n=132: `title x3` даёт MRR 0.5354 против 0.5212, `title x3 tags x2` — 0.5382, дальше спад (`title x8` — 0.5207, уже хуже базы). Максимум сдвига — 3 запроса из 132 при McNemar −1/+4. Это шум.
- **Проверено и отвергнуто «выбросить семантику», хотя в агрегате оно выглядело победой.** Ряд по весу BM25 монотонен и соблазнителен: x1 → x2 → x3 → x6 → только BM25 даёт MRR 0.5212 → 0.5435 → 0.5431 → 0.5442 → **0.5493**, recall@1 0.3939 → 0.4470. Но бутстрэп (10 000 выборок) показывает ΔMRR +0.0281 с ДИ **[−0.0158, +0.0738]** — ноль внутри интервала, а разбор по запросам объясняет механику: при отключённой семантике **хуже стало 27 запросам, а лучше 25**. Прирост среднего держится на нескольких крупных улучшениях, перекрывающих чуть более многочисленные ухудшения — то есть ровно та конфигурация, которая выигрывает на бумаге и проигрывает в работе. Промежуточный `x2` (18 лучше / 14 хуже, ДИ [−0.0008, +0.0466]) на границе, но тоже не проходит.
- ⚠️ **Узкое место сместилось с идей на мощность замера.** При n=132 различимы только сдвиги примерно от **0.024 MRR**, тогда как весь остаток (поднять все 75 «найденных, но не первых» на первое место) — это 0.31 MRR. То есть шаг измеримости ≈ 8% от доступного потолка: любое аккуратное улучшение ранжирования будет тонуть в шуме. Наращивать набор важнее, чем перебирать веса; ближайший резерв — окно по времени в `build_golden` (см. v1.32.0), которое сейчас и добавляет мусор, и не даёт использовать пары с большим разрывом.

### Chore

- +2 теста (532 всего): дефолтные веса нейтральны; нулевой вес канала действительно исключает документ, найденный только этим каналом.

## v1.32.0 — 2026-07-19

Разбор «недостижимых 13.6%», оставшихся после v1.31.0. Ни одного дефекта поиска там не оказалось: остаток целиком объясняется **мусором в ground truth**. Заодно исправлен дефект харнесса, из-за которого часть метрики была зафиксированным нулём независимо от кода.

### Fixed

- **`build_golden` требовал от поиска недостижимого.** Набор считает «ответом» всё, что открыли после запроса и до следующего. Но запрос обычно ограничен проектом, а открыть можно что угодно: ассистент ищет одно, потом лезет в статью ДРУГОГО проекта за кредами — и это засчитывается как правильный ответ на предыдущий запрос. Замер на живом логе: **35 пар из 299 (11.7%) указывали вне скоупа**, а у **8 запросов из 140 (5.7%) недостижимы были ВСЕ ожидания** — эти 5.7% метрики не зависели от качества поиска вообще. Новая `filter_reachable()` отсеивает такие пары. Отсев независим от ранжирования (опирается на скоуп запроса, а не на то, нашлась ли статья) — иначе получилось бы «выкинем то, что не находится», и метрика росла бы сама от себя.
- **Категория диагностики «срезана порогом» смешивала два разных диагноза** — цель, убранную отсечкой по скору, и цель, просто лежащую ниже десятого места в пуле. На замере это выглядело как «виноват порог» у 25 запросов; после расщепления оказалось, что **порог не виноват НИ РАЗУ** — у всех 25 цель была на 11-м месте и глубже, то есть это чистое ранжирование. Категории разделены (`3_срезана_порогом` / `4_глубоко_в_слиянии`).

### Added

- **`load_golden()` — единая точка сборки набора** (разбор лога → живые статьи → достижимые). Пять скриптов оценки собирали набор каждый своей строчкой; любое уточнение метода пришлось бы разносить по всем пяти, где оно неизбежно разъехалось бы — и конфигурации сравнивались бы на разных наборах. `in_scope=None` сохраняет прежнее поведение для сверки с историческими цифрами.

### Changed

- ⚠️ **Опубликованный baseline v1.31.0 занижен харнессом, а не кодом.** Те же самые сборка и код на очищенном наборе: **MRR 0.4914 → 0.5212, recall@1 0.3714 → 0.3939, recall@10 0.7286 → 0.7727** (n=140 → 132). Поведение поиска НЕ менялось. Контроль, что это не подгонка: прежний конвейер на том же очищенном наборе даёт MRR 0.4499 против 0.5212 — выигрыш v1.31.0 сохраняется целиком (+0.0713), значит он не был артефактом грязного набора. **Сравнивать цифры между версиями можно только при одинаковой версии харнесса.**
- **Диагностика приведена к боевой конфигурации** — группировка термов, скоуп и ширина пула берутся из `SEARCH_*`, а не захардкожены. Это корректирует историческую запись v1.7.x, где reranker оценивался как «+25–40% precision при ~50ms overhead»: на данном железе и корпусе overhead оказался в ~300 раз выше заявленного, а прирост неизмерим.
- **Флаг `RERANK_ENABLED` (дефолт `false`)** вместо обнуления бюджета. Выставить `SEARCH_RERANK_BUDGET_S=0` было бы недостаточно: `wait_for` снимает ожидание, но фоновый поток всё равно досчитывает `predict` и жжёт те же ~14.5 с CPU впустую. При выключенном флаге `_rerank_async` выходит сразу, не запуская поток, а прогрев моделей на старте не грузит reranker вовсе (экономия RSS и времени старта). Обратно включается `RERANK_ENABLED=1` — осмысленно только вместе с бо́льшим `SEARCH_CANDIDATE_POOL` и/или лёгкой моделью и обязательно с повторным замером тем же харнессом. +4 теста, два существующих теста интеграции reranker'а приведены к новой семантике (включают флаг явно).
- **Уточнён baseline из v1.26.0.** Опубликованные там цифры (n=40: MRR 0.538, recall@10 0.70) сняты на выборке из 40 ПОСЛЕДНИХ запросов и оказались оптимистичными — свежие запросы легче. Честная точка отсчёта по всему набору: **n=132, MRR 0.4634, recall@10 0.6515**.

## v0.12.0 (pre-release) — 2026-04-15

Search quality + UX. 36 MCP tools, 49 тестов.

### Поиск

- **Cross-encoder reranking** — после hybrid retrieval (BM25 + semantic + decay) топ-20 кандидатов пересортируются `BAAI/bge-reranker-base` (280MB, multilingual, lazy load). Precision@3 +15-20%. Graceful degradation при ошибке загрузки модели.
- **start_task релевантность** — все блоки фильтруются по теме: search >= score 15, active_context только пересекающиеся записи, session показывается только при совпадении слов, deps — только релевантные. Раньше выгружались последние 10 действий и полная сессия независимо от темы.

### Web UI

- **Snippets с подсветкой** — `/api/search` возвращает `snippets` (строки с совпадениями + контекст ±1 строка, max 5 на статью). UI рендерит monospace-блоки вместо общего preview. Слова запроса подсвечиваются `<mark>` жёлтым в title, snippets и развёрнутой статье.
- **Auto-scroll** к первому совпадению при expand статьи.
- **Расшифровка ENC: на лету** — поиск работает по содержимому секретов (только для авторизованных).
- **Счётчик совпадений** в meta строке карточки.

### Граф

- **Top-K edges per node** — после Obsidian-импорта (209 статей) граф имел 10725 связей ("волосяной шар"). Теперь max 8 strongest связей на узел → 650 edges, читаемо.
- **Orphan marker** — узлы без связей подсвечиваются серым (50% opacity, меньший размер) — честный сигнал "статья изолирована".
- **Live PROJECTS list** — `web_graph/analytics/tags` теперь вызывают `_discover_projects()` на каждый запрос. Раньше использовали кэш PROJECTS — проекты, созданные в других процессах (docker exec), не появлялись в графе до рестарта.

### Прочее

- **Stats fix** — `tools.py` инкрементировал счётчик только для 5 legacy ключей. Теперь учитываются все 36 tools.
- **PostToolUse hook matcher** — расширен с `(save_lesson|finish_task)` до полного списка: `save_decision`, `save_runbook`, `save_from_template`, `save_secret`, `ingest`, `import_obsidian`, `git_capture`, `edit_article`.
- **CLAUDE.md правила** — добавлены таблицы выбора проекта (9 проектов) и tool (8 типов).
- **Project deps** — настроены примерные зависимости между проектами.
